# Internationalization

Three layers of i18n, each with its own mechanism.

## Layer 1: UI Strings

JSON files in `src/i18n/` (`en.json`, `fr.json`, `es.json`) with the `t()` helper function. Standard key-value translation for interface text.

## Layer 2: URL Path Translations

Translatable URL segments (e.g., `/routes/` → `/parcours/` in French) handled by:

- **`src/lib/i18n/segment-registry.ts`** — static map of segment translations. This is a `const`, NOT a mutable variable set at runtime. Vite's SSR bundle runs in a separate module graph from Astro integration hooks, so module-level mutable state set during `astro:config:setup` would not be available at render time.
- **`src/lib/i18n/path-translations.ts`** — `translatePath()` and `reverseTranslatePath()` functions
- **`src/integrations/i18n-routes.ts`** — defines locale page routes, uses `translatePath()` for pattern generation
- **`src/lib/i18n/locale-switcher.ts`** — `switchLocalePath()` computes the URL for switching between locales, handling reverse-translation from current locale then translation to target

**Sync requirement:** Route patterns in `i18n-routes.ts` and segment translations in `segment-registry.ts` MUST stay in sync. Adding a route with a translatable segment without a corresponding entry in `segment-registry.ts` means the French/Spanish URL uses the English segment.

## Layer 3: Content Sidecar Files

Content translations use sidecar files: `index.fr.md` next to `index.md`. Translation files (`*.??.md`) are excluded from base collection loading and loaded separately by custom loaders.

## Tag Translations

`tTag()` in `src/lib/i18n/tag-translations.ts` translates route tags (e.g., "riverside" → "riviere") using `{cityDir}/tag-translations.yml`. This file is a build-time transform — `build-data-plugin.ts` replaces it with a static map. If you change its exports, update the transform.

## Locale Utilities

`src/lib/i18n/locale-utils.ts` provides:

- `shortLocale()` / `fullLocale()` — convert between `"en"` and `"en-CA"`
- `defaultLocale()` — returns the short locale (e.g., `"en"`, not `"en-CA"`)
- `supportedLocales()` — city-aware locale list
- `localeLabel()` — human-readable locale name

Locales are derived from the city's `config.yml` (e.g., `[en-CA, fr-CA]` becomes `[en, fr]`). Never hardcode locale values.

Use `fullLocale()` when you need the full BCP47 tag for `Intl` formatting.
